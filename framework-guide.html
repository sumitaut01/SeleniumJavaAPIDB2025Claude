System Design

Architecture Overview

Three independent layers connected by a single thread-local bridge called TestContext. Each layer can run alone or be combined with others in any order within a single test method.

💡
The Key Insight TestContext is the thread-local state bag that lets any layer communicate with any other layer in the same test. API writes a userId. DB reads it to validate the record. UI reads it to fill a login form. All in one test method, zero coupling between layers.

Getting Started

Quick Start

Import the project, fill in your env-config.json URLs, and run your first test in under 5 minutes.

  1. 1
    Clone & import into IntelliJ / Eclipse
    File → Open → select the pom.xml. Maven will download all 15 dependencies automatically.
  2. 2
    Fill in env-config.json
    Edit src/test/resources/config/env-config.json. Replace placeholder URLs and DB credentials with your real ones. Set oracle: null for any country that doesn't use Oracle.
  3. 3
    Add your locators to LoginPage.java
    Replace id="username" etc. in LoginPage.java with your application's actual selectors.
  4. 4
    Run the sample tests
    mvn test -Denv=qa -Dcountry=IN
    Chrome opens, tests run, ExtentReport HTML is generated in reports/.
  5. 5
    Write your first test
    Create a class extending BaseTest (UI), BaseApiTest (API), or BaseHybridTest (both). See the Patterns section for templates.
Run Commands
bash Maven CLI — all common scenarios 
# ── Defaults: env=qa, country=IN, local chrome ──────────────

# ── Specific environment and country ────────────────────────

mvn test -Dgroups=api

# ── Run only DB tests (no browser, no API calls) ────────────
mvn test -Dgroups=db

# ── Run full-stack hybrid tests ──────────────────────────────
mvn test -Dgroups=fullflow

# ── Smoke across all layers ──────────────────────────────────
mvn test -Dgroups=smoke -Denv=qa -Dcountry=IN

# ── Disable retries (useful for debugging) ───────────────────
mvn test -Dretry.count=0

Layer 1

UI Layer — Selenium

Browser automation using Selenium 4 + WebDriverManager. All page interactions go through Page Objects that extend BasePage. Tests never call WebDriver directly.

How to Write a Page Object
java pages/DashboardPage.java — extend BasePage 
public class DashboardPage extends BasePage {

  // ── @FindBy locators (PageFactory) ───────────────────────────
  @FindBy(css = ".welcome-banner")
  private WebElement welcomeBanner;

  @FindBy(id = "logoutBtn")
  private WebElement logoutBtn;

  // Dynamic locators use By.*
  private final By orderRow = By.cssSelector("[data-order-id='%s']");

  // ── Actions ───────────────────────────────────────────────────
  public DashboardPage open() {
    navigateTo(config().getBaseUrl() + "/dashboard");
    return this;
  }

  public String getWelcomeMessage() {
    return getText(By.cssSelector(".welcome-banner"));
  }

  public boolean isOrderVisible(String orderId) {
    return isDisplayed(By.cssSelector("[data-order-id='" + orderId + "']"));
  }

  // ── RULE: return raw values — never assert inside pages ───────
  public String getTitle()      { return getPageTitle(); }
  public String getCurrentUrl() { return getCurrentUrl(); }
}
BasePage Helper Methods
MethodWhat it does
click(By locator)Waits for element to be clickable, then clicks
type(By locator, String text)Clears field, waits for visible, types text
getText(By locator)Waits for visible, returns trimmed text
isDisplayed(By locator)Returns true/false — safe, no exception if missing
navigateTo(String url)Navigates driver to URL
jsClick(WebElement)JavaScript click — for elements that intercept events
scrollIntoView(WebElement)Scrolls element into viewport
driver()Returns current thread's WebDriver (rarely needed)
📏
Page Object Rule Pages only perform actions and return raw values. They never contain assertThat() or Assert.assertEquals(). All assertions belong in the test class.
Layer 2

API Layer — RestAssured

RestAssured wrapped in a clean service/builder/validator pattern. Domain services (AuthService, UserService) mirror the Page Object pattern — test classes never call RestAssured directly.

java RequestBuilder — fluent spec factory 
// Full chain
RequestSpecification spec = RequestBuilder.create()
    .withAuth()              // Bearer token from TestContext — automatic
    .withCountryHeader()     // adds X-Country-Code: IN
    .withQueryParam("page", 1)
    .withQueryParam("size", 20)
    .relaxedHttps()          // ignores self-signed certs in non-prod
    .build();

// With explicit token (not from context)
RequestSpecification spec2 = RequestBuilder.create()
    .withAuth("some-other-token")
    .withHeader("X-Correlation-ID", UUID.randomUUID().toString())
    .withBody(myRequestPojo)
    .build();

// Then execute
Response response = ApiClient.post(spec, ApiEndpoints.User.CREATE);
java ResponseValidator — fluent assertion chain 
ResponseValidator.of(response)
    .statusCode(201)
    .contentTypeIsJson()
    .hasField("data.userId")            // field exists and not blank
    .fieldEquals("data.status", "ACTIVE")  // exact value match
    .fieldContains("data.email", "@myapp") // contains substring
    .listNotEmpty("data.roles")           // array is not empty
    .responseTimeBelow(3000)              // under 3 seconds
    .matchesSchema("schemas/user-response.json") // JSON Schema
    .validate();                           // runs all soft assertions

// Extract values for downstream use
UserResponse user = response.jsonPath().getObject("data", UserResponse.class);
String token = ResponseValidator.of(response).extractString("accessToken");

Layer 3

DB Layer — JDBC + HikariCP

Two database clients (Postgres + Oracle) managed through HikariCP connection pools. Three assertion styles: fluent validator, typed mapper, or generic map. All SQL lives in DbQueries — never inline in test code.

java DbValidator — fluent assertion chain (preferred style) 
// Style 1: fluent chain via dbAssert() shortcut in BaseTest
dbAssert(pgDb)
    .forQuery(DbQueries.User.FIND_BY_EMAIL, email)
    .rowExists()                         // ≥ 1 row returned
    .columnEquals("status", "ACTIVE")    // first row, column = value
    .columnEquals("country", country())  // uses active test country
    .columnNotBlank("user_id")          // not null and not empty
    .columnIsNull("deleted_at")         // IS NULL in DB
    .validate();                         // runs all soft assertions

// Count assertions
dbAssert(pgDb)
    .forQuery(DbQueries.Order.COUNT_BY_USER, userId)
    .scalarEquals(1L)                    // COUNT(*) = 1
    .validate();

// Multi-row assertions
dbAssert(pgDb)
    .forQuery(DbQueries.Order.FIND_BY_USER, userId)
    .rowCountAtLeast(1)                  // at least 1 order
    .rowCountEquals(3)                   // exactly 3 orders
    .validate();

// Oracle (same API, different client)
if (orDb != null) {
    dbAssert(orDb)
        .forQuery(DbQueries.Audit.COUNT_EVENTS_BY_ENTITY, userId, "USER_CREATED")
        .scalarEquals(1L)
        .validate();
}

Both clients in BaseTest pgDb and orDb are automatically created in BaseTest.setUp() if the DB is configured for the active env+country. orDb will be null if Oracle is not configured — always guard with if (orDb != null).
java RowMapper — typed POJO from ResultSet 
// Style 2: typed mapper → get a strongly-typed POJO back
Optional<UserDbRecord> result = pgDb.queryOne(
    DbQueries.User.FIND_BY_EMAIL,
    UserDbRecord.rowMapper(),   // built-in mapper on the POJO class
    email);

UserDbRecord user = result.orElseThrow();
assertThat(user.getStatus()).isEqualTo("ACTIVE");
assertThat(user.getEmail()).isEqualTo(expectedEmail);

// Writing your own RowMapper for a new table
public class ProductDbRecord {
    public static RowMapper<ProductDbRecord> rowMapper() {
        return rs -> ProductDbRecord.builder()
            .productId(rs.getString("product_id"))
            .name(rs.getString("name"))
            .price(rs.getBigDecimal("price"))
            .build();
    }
}

// Style 3: generic map (no POJO — good for one-off queries)
List<Map<String, Object>> rows = pgDb.query(
    "SELECT * FROM audit_log WHERE entity_id = ?", userId);
String evt = (String) rows.get(0).get("event_type"); // keys always lowercase
java DbQueries — central SQL registry 
// ALL SQL lives in DbQueries.java — grouped by domain
public final class DbQueries {

    public static final class User {
        public static final String FIND_BY_EMAIL =
            "SELECT user_id, email, status, country FROM users WHERE email = ?";
        public static final String EXISTS_BY_EMAIL =
            "SELECT COUNT(*) FROM users WHERE email = ?";
        public static final String UPDATE_STATUS =
            "UPDATE users SET status = ?, updated_at = NOW() WHERE user_id = ?";
        // ... more queries
    }

    public static final class Order {
        public static final String COUNT_BY_USER =
            "SELECT COUNT(*) FROM orders WHERE user_id = ?";
        // ... more queries
    }

    public static final class Audit {
        // Oracle-specific variant (no LIMIT — uses ROWNUM)
        public static final String LATEST_EVENT_ORA =
            "SELECT event_type FROM audit_log WHERE entity_id = ? AND ROWNUM = 1";
    }
}
📏
Rule: No Inline SQL Never write SQL strings directly in test classes. All queries go in DbQueries.java. For Oracle-specific syntax, add a _ORA constant alongside the Postgres one.

Advanced Patterns

Hybrid Flows — All Layers Together

The most powerful tests in the suite. A single test method validates the same business action from all three angles: API response, database state, and browser behaviour.

java Full 3-layer test: API creates user → DB verifies → UI confirms login 
public class FullFlowTest extends BaseHybridTest {

  @Test(groups = "fullflow")
  public void testCreateViaApi_VerifyDb_LoginUI() {

    // ── Phase 1: API ─────────────────────────────────────────────
    authService.loginExpectSuccess("admin@qa.in", "Admin@1234");
    UserResponse user = userService.createUserExpectSuccess(
        userService.buildUniqueUserRequest());
    // TestContext now has: authToken, userId, createdEmail

    // ── Phase 2: DB — Postgres ───────────────────────────────────
    dbAssert(pgDb)
        .forQuery(DbQueries.User.FIND_BY_EMAIL, user.getEmail())
        .rowExists()
        .columnEquals("status",  "ACTIVE")
        .columnEquals("country", country())
        .columnIsNull("deleted_at")
        .validate();

    // ── Phase 2B: DB — Oracle audit ──────────────────────────────
    if (orDb != null) {
        dbAssert(orDb)
            .forQuery(DbQueries.Audit.COUNT_EVENTS_BY_ENTITY,
                      user.getUserId(), "USER_CREATED")
            .scalarEquals(1L)
            .validate();
    }

    // ── Phase 3: UI — browser login ──────────────────────────────
    LoginPage login = loginViaUI(ctx().getCreatedEmail(), "Test@1234");
    // loginViaUI reads createdEmail from TestContext automatically

    assertThat(login.isLogoutLinkVisible())
        .as("User created via API should be able to log in")
        .isTrue();
  }
}
java DB as source of truth: pre-condition → API action → post-condition 
@Test(groups = "fullflow")
public void testDbPreCondition_ApiAction_DbPostCondition() {

  // Setup: create a clean user via API
  authService.loginExpectSuccess("admin@qa.in", "Admin@1234");
  UserResponse user = userService.createUserExpectSuccess(
      userService.buildUniqueUserRequest());

  // ── Pre-condition: DB confirms 0 orders ─────────────────────
  dbAssert(pgDb)
      .forQuery(DbQueries.Order.COUNT_BY_USER, user.getUserId())
      .scalarEquals(0L)
      .validate();

  // ── API action: place an order ───────────────────────────────
  orderService.createOrder(Map.of("productId", "PROD-001", "quantity", 1));

  // ── Post-condition: DB confirms order was created ────────────
  dbAssert(pgDb)
      .forQuery(DbQueries.Order.COUNT_BY_USER, user.getUserId())
      .scalarEquals(1L)      // count went from 0 → 1
      .validate();

  dbAssert(pgDb)
      .forQuery(DbQueries.Order.FIND_BY_ID, ctx().getOrderId())
      .rowExists()
      .columnEquals("status", "PENDING")
      .columnEquals("user_id", user.getUserId())
      .validate();
}
java Async flow — polling until DB catches up (Kafka / event-driven)
// Scenario: API returns 201 immediately, but a Kafka consumer // processes the event and writes to DB asynchronously. // AwaitilityUtil polls the DB until the row appears. @Test(groups = "fullflow") public void testAsyncOrderProcessing() { // Place order via API (returns 201 immediately) authService.loginExpectSuccess("admin@qa.in", "Admin@1234"); userService.createUserExpectSuccess(userService.buildUniqueUserRequest()); orderService.createOrder(Map.of("productId", "PROD-001")); // Wait up to 30s for the Kafka consumer to write the order to DB AwaitilityUtil.waitUntilDbRowExists(pgDb, "SELECT COUNT(*) FROM orders WHERE user_id = ?", ctx().getUserId()); // Now assert — the row definitely exists dbAssert(pgDb) .forQuery(DbQueries.Order.FIND_BY_ID, ctx().getOrderId()) .rowExists() .columnEquals("status", "PENDING") .validate(); // Wait for status to transition to PROCESSING AwaitilityUtil.waitUntilDbValueEquals(pgDb, "SELECT status FROM orders WHERE order_id = ?", "PROCESSING", ctx().getOrderId()); }

Automation

Listeners — What Runs Automatically

Four TestNG listeners registered globally in testng.xml. You never call them directly — they run automatically around every test.

ListenerWhen it firesWhat it does
RetryTransformer Before suite starts Injects RetryAnalyzer into every @Test globally. Controls retries via -Dretry.count=N
TestListener Every test lifecycle event onStart: creates ExtentTest node. onFail: logs exception + captures screenshot. onFinish: flushes report
SoftAssertListener After each passing test Scans for un-flushed SoftAssertions fields. Logs warning if assertAll() was never called on a passing test
ExtentReportApiFilter Every API call via RequestBuilder Auto-embeds full request (method, URI, headers, body) and response (status, time, body) into the current test's ExtentReport node
⚠️
RetryTransformer must be listed FIRST in testng.xml It's an IAnnotationTransformer — TestNG processes it at test collection time, before any test runs. If placed after TestListener, retries won't be applied.

Best Practices

Golden Rules

These rules keep the framework clean and maintainable as the team grows. Bookmark this page.

  1. ①
    No inline SQL in test classes. All SQL goes in DbQueries.java. Oracle variants get a _ORA suffix. This is the single-point-of-change when your schema evolves.
  2. ②
    No inline URL strings in test classes. All API paths go in ApiEndpoints.java. All web URLs come from config().getBaseUrl(). Never hardcode a URL.
  3. ③
    Page Objects never assert. Pages return raw values (strings, booleans). Assertions with assertThat() or SoftAssertions belong in the test class only.
  4. ④
    Always call soft.assertAll(). AssertJ's SoftAssertions collect failures silently — they only throw when you call assertAll(). Forgetting it means your test passes even with assertion failures. The SoftAssertListener will warn you, but don't rely on it.
  5. ⑤
    Guard Oracle with if (orDb != null). Not every env/country has Oracle. orDb is null when Oracle isn't configured. Always check before using it, or use dbHelper.auditEventCount() which handles null safely.
  6. ⑥
    Clean up test data in @AfterMethod. Tests that create DB records must delete them. Store the email/ID in an instance variable, call dbHelper.cleanupTestUser(email) in @AfterMethod(alwaysRun=true). Never leave junk data in QA/UAT DBs.
  7. ⑦
    Use BaseApiTest for DB-only tests. There's no need to start a browser for pure DB validation. BaseApiTest skips driver initialisation, making DB-only tests much faster to run.
  8. ⑧
    One TestContext per thread — don't share across tests. TestContext is initialised and cleared by BaseTest per test method. Never store a reference to it at the class level or pass it to another thread.
🎯
Adding a new feature to the framework? New DB query → DbQueries.java. New API endpoint → ApiEndpoints.java. New constant → FrameworkConstants.java. New country → Country.java enum + env-config.json. New page → extend BasePage. New API service → inject RequestBuilder + ApiClient.
